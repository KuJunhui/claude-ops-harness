# claude-ops-harness

Java/Spring 프로젝트 운영 중 발생하는 문제를 Claude Code로 **분석부터 배포까지 자동화**하는 하네스입니다.

> 대상 프로젝트에서 `.claude/settings.local.json`은 `.gitignore`에 등록되어 커밋되지 않으며, 나머지 `.claude/` 파일과 `CLAUDE.md`는 공유됩니다.

## 포함 파일

| 파일 | 설명 |
|------|------|
| `.claude/settings.json` | 프로젝트 레벨 설정 (hooks 등) |
| `.claude/commands/problem.md` | `/problem` 커스텀 명령어 — 문제 분석부터 배포까지 원스톱 처리 |
| `.claude/commands/ship.md` | `/ship` 커스텀 명령어 — 코드 변경 배포 자동화 |
| `.claude/common/pipeline.md` | `/problem`, `/ship` 공통 배포 파이프라인 (커밋 → PR → CI → main 머지 → CD) |
| `.claude/hooks/validate-git-sensitive.sh` | PreToolUse hook — git 명령에서 민감 파일(`.env`, `.claude/settings.local.json` 등) 커밋 차단 |

## 설치 방법

### 1. 이 레포 클론

```bash
# 대상 프로젝트와 같은 상위 디렉토리에 클론하는 것을 권장합니다.
git clone https://github.com/KuJunhui/claude-ops-harness.git
```

### 2. 설치 스크립트 실행

```bash
# 대상 프로젝트 루트에서 실행
cd /path/to/your-project
/path/to/claude-ops-harness/install.sh
```

예시 (상위 디렉토리에 나란히 클론한 경우):

```bash
cd ~/projects/my-spring-app
../claude-ops-harness/install.sh
```

설치가 완료되면 아래 파일들이 대상 프로젝트에 복사됩니다:

```
your-project/
└── .claude/
    ├── settings.json
    ├── VERSION
    ├── commands/
    │   ├── problem.md
    │   └── ship.md
    ├── common/
    │   └── pipeline.md
    └── hooks/
        └── validate-git-sensitive.sh
```

### 3. Claude Code 실행

```bash
cd /path/to/your-project
claude
```

## 업데이트 방법

하네스 설정이 변경되면:

```bash
cd /path/to/claude-ops-harness
git pull

cd /path/to/your-project
/path/to/claude-ops-harness/install.sh
```

설치 스크립트를 다시 실행하면 최신 설정으로 덮어씁니다.

## `/problem` 사용법

문제 분석부터 배포까지 원스톱으로 처리하는 커스텀 명령어입니다.

> **참고**: `/problem`은 메인 에이전트 **Opus** 기준으로 작성되었습니다.

### 실행 방법

**방법 1: 간단한 문제** — 에러 로그나 문제 내용을 `/problem` 뒤에 바로 붙여넣기

```
/problem [에러 로그 붙여넣기 or 문제 내용]
```

**방법 2: 복잡한 문제** — 먼저 Claude Code와 대화하며 문제와 해결책을 구체화한 뒤 `/problem` 실행

```
# 1. 먼저 대화로 문제 파악
> 로그인 후 토큰 갱신이 안 되는 것 같은데 원인이 뭘까?
> (Claude와 대화하며 원인 분석 및 해결 방향 정리)

# 2. 정리가 끝나면 실행
> /problem
```

### 사용자 개입 포인트

프로세스 진행 중 사용자가 직접 개입하는 구간은 **2곳**입니다:

1. **해결책 선택** (Phase 1) — Claude가 제시한 해결책 중 적용할 번호를 선택
2. **최종 확인** (Phase 5) — 모든 코드 수정 및 테스트 완료 후 결과 확인 → 추가 수정이 필요하면 요청, 문제없으면 OK로 다음 단계(커밋·PR·배포) 진행

이 2곳을 제외한 나머지 과정(브랜치 생성, 코드 변경, 테스트, PR, CI/CD, 배포)은 자동으로 진행됩니다.

## `/ship` 사용법

코드 수정이 이미 완료된 상태에서, 검증과 배포만 자동으로 처리하는 커스텀 명령어입니다. `/problem`과 달리 문제 분석이나 코드 변경을 수행하지 않습니다.

> **참고**: `/ship`은 메인 에이전트 **Opus** 기준으로 작성되었습니다.

### 실행 방법

**방법 1: 자동 분석** — 타입과 설명을 Claude가 변경사항에서 추론

```
/ship
```

**방법 2: 타입·설명 직접 지정** — 사용자 확인 없이 바로 검증 단계로 진행

```
/ship feat 알림 설정 API 추가
```

### dev 브랜치 vs 이슈 브랜치

| 현재 브랜치 | 동작 |
|-------------|------|
| `dev` | 변경사항 분석 → 검증 → 이슈·브랜치 자동 생성 → 배포 (전체 흐름) |
| 이슈 브랜치 (`feat/#123` 등) | 이슈·브랜치 생성을 스킵하고 검증 → 배포로 직행 |

### 사용자 개입 포인트

dev 브랜치에서 타입·설명 없이 실행한 경우 **1곳**:

1. **변경사항 확인** (Phase 1) — Claude가 분석한 타입과 설명을 확인하고, 필요시 수정 요청

타입·설명을 인자로 전달하거나, 이슈 브랜치에서 실행하면 사용자 개입 없이 자동으로 진행됩니다.

## 설계 의도

### `/problem` 워크플로우 설계 결정

#### 1. API / 비API 작업 분류

하나의 해결책 안에서도 작업을 `[API]`(Controller/Service/Repository 레이어)와 `[비API]`(설정, 인프라, 유틸)로 분류합니다.

**이유**: API 레이어 변경은 비즈니스 로직에 직접 영향을 주므로 반드시 Test-First로 검증해야 합니다. 반면 설정 변경이나 인프라 수정에 불필요한 테스트를 강제하면 오버헤드만 늘어납니다. 작업 성격에 따라 절차를 분리함으로써 **안전성과 속도를 동시에 확보**합니다.

#### 2. Test-First 강제

`[API]` 작업은 반드시 테스트를 먼저 작성하고, 테스트 실패 시 구현 코드를 수정합니다 (테스트가 아닌).

**이유**: AI가 코드를 작성할 때 가장 큰 위험은 "돌아가는 것 같지만 실제로는 스펙과 다른 코드"입니다. 테스트를 먼저 작성하면 **테스트가 명세(spec) 역할**을 하므로, AI가 생성한 구현이 의도한 동작과 일치하는지 자동으로 검증됩니다. 테스트 실패 시 테스트가 아닌 구현을 수정하는 규칙은 이 명세 역할을 보호합니다.

#### 3. 병렬 워크트리 실행

독립적인 작업 그룹을 Agent tool의 `isolation: "worktree"`로 동시에 실행하고, 완료 후 cherry-pick으로 통합합니다.

**이유**: 서로 다른 도메인의 변경을 순차적으로 처리하면 불필요하게 시간이 늘어납니다. Git worktree를 활용하면 **각 작업이 완전히 격리된 환경에서 독립 실행**되므로, 상호 간섭 없이 병렬 처리가 가능합니다. 통합 시 cherry-pick을 사용하는 이유는 각 워크트리의 커밋을 선택적으로 가져올 수 있고, 충돌 발생 시 메인 에이전트가 컨텍스트를 보고 판단할 수 있기 때문입니다.

#### 4. Sonnet → Opus 에스컬레이션

CI/CD 파이프라인이 정상일 때는 서브에이전트를 Sonnet으로, 실패가 발생하면 Opus로 전환합니다.

**이유**: 커밋, PR, 머지, CI 폴링 같은 정형화된 작업은 Sonnet으로 충분합니다. 하지만 CI 실패 분석, 충돌 해결, CD 로그 디버깅은 **고차원적 판단이 필요한 비정형 작업**이므로 Opus가 적합합니다. 이 전략으로 **비용을 절감하면서 실패 대응 품질을 유지**합니다.

#### 5. CI 검증 없는 배포 금지

코드 수정이 포함된 재시작은 반드시 CI (Phase 7a)를 다시 거치도록 강제합니다.

**이유**: 파이프라인 실패 후 "빠르게 고쳐서 바로 배포"하는 유혹이 있지만, 이는 2차 장애의 원인이 됩니다. 자동화된 워크플로우일수록 **안전 게이트를 건너뛰지 않는 것**이 중요합니다. 재시작 매핑 테이블은 어떤 실패 시점에서든 어디로 돌아가야 하는지 명확하게 정의하여 혼란을 방지합니다.

#### 6. 파괴적 DB 마이그레이션 2단계 배포

파괴적 스키마 변경(`DROP`, `RENAME COLUMN`, `ADD NOT NULL` 등)이 포함된 작업은 1차 배포(앱 코드 변경)와 2차 배포(파괴적 마이그레이션)로 분리합니다.

**이유**: Flyway 마이그레이션은 앱 이미지를 롤백해도 되돌아가지 않습니다. 파괴적 마이그레이션을 앱 코드와 한 번에 배포하면, 롤백 시 Hibernate `validate`가 실패하여 **앱이 시작조차 불가능한 상태**가 됩니다. 2단계로 분리하면 1차 배포 후 안정성을 확인한 뒤 스키마를 변경하므로, 롤백 가능성을 유지하면서 안전하게 마이그레이션할 수 있습니다.

#### 7. 환경변수 추가 체크리스트

새 환경변수 도입 시 전달 경로(GitHub Secrets → CD workflow → SSH → Docker Compose → 컨테이너 → Spring) 전체를 체크리스트로 검증합니다.

**이유**: 환경변수 전달 경로는 여러 파일에 걸쳐 있어 한 곳이라도 누락되면 프로덕션에서 NPE나 시작 실패가 발생합니다. 특히 AI 에이전트가 앱 코드에만 환경변수를 추가하고 CD 워크플로우나 Docker Compose를 빠뜨리는 실수가 반복되었습니다. 체크리스트를 Phase 5 통합 검토에 포함시켜 **배포 전에 누락을 체계적으로 방지**합니다.

#### 8. 민감 파일 커밋 차단 Hook

PreToolUse hook으로 `.env`, `.claude/settings.local.json` 등의 민감 파일이 git에 추가되는 것을 사전 차단합니다.

**이유**: AI 에이전트가 `git add -A`를 실행하면 의도치 않게 시크릿이나 로컬 설정이 커밋될 수 있습니다. 이 hook은 **자동화의 편의성을 유지하면서 보안 사고를 예방**합니다.
