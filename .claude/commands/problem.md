# /problem — 문제 분석부터 배포까지 원스톱 처리

문제 상황을 분석하여 해결책을 제시하고, 사용자가 선택한 해결책을 실행하여 배포까지 완료한다.
브랜치명에 `#`이 포함되므로, 모든 git/gh 명령에서 브랜치명은 반드시 따옴표로 감싼다.

> **빌드/테스트 명령 표기**: 이 문서는 특정 빌드 도구에 종속되지 않도록 명령을 추상적으로 표기한다. "컴파일 검증", "대상 테스트 실행", "전체 빌드+테스트" 등은 프로젝트의 빌드 도구(Gradle/Maven 등)에 맞는 실제 명령으로 치환하여 실행한다. 프로젝트 루트의 빌드 스크립트를 먼저 확인해 명령을 파악한다.

## Preflight

1. `gh auth status` — 인증 실패 시 **중단**
2. `git status --short` — 비어 있지 않으면 기존 변경사항을 임시 보관한다:
   - `git stash push -u -m "problem-preflight-stash"` (untracked 파일 포함)
   - `HAS_STASH=true` 플래그 설정 (비어 있으면 `HAS_STASH=false`)
3. `git checkout dev && git pull origin dev` — 항상 최신 dev에서 시작
4. `HAS_STASH=true`이면 기존 변경사항을 dev 위에 복원한다:
   - `git stash pop` — 충돌 발생 시:
     - 해결 전 `git stash show -p --include-untracked "stash@{0}"`를 `.problem/local/preflight-stash.patch`로 백업한다 (`-u`로 stash한 untracked 파일까지 포함해야 데이터 손실이 없다. `--include-untracked`는 git 2.32+ — 미지원 구버전이면 `git show "stash@{0}^3"`을 별도 파일로 병기 백업한다. `.problem/local/`은 커밋 금지 로컬 전용 디렉토리 — 없으면 생성한다)
     - 충돌 파일을 분석하여 자체 해결한 뒤 진행하고, 해결 내역(어느 파일을 어떻게 병합했는지)을 사용자에게 보고한다
   - 이후 Phase 1~6은 기존 변경사항이 워킹 트리에 있는 상태에서 진행된다 (Phase 3에서 WIP 커밋으로 승격되어 워킹 트리는 clean 상태가 된다)

## Phase 1: 문제 분석 & 해결책 제시

> **단순 변경 스킵**: 사용자가 구체적인 변경 내용을 직접 명시한 경우 (예: "토큰 만료 시간 30분으로 변경해줘") Phase 1을 스킵하고 해당 내용을 단일 작업으로 구성하여 Phase 2로 직행한다.

1. 사용자가 제시한 에러 로그 / 문제 상황을 분석한다.
2. 관련 코드를 탐색하여 원인을 파악한다.
3. 해결책을 번호와 함께 제시한다. 하나의 해결책 안에 API 변경과 비API 변경이 섞일 수 있으므로, **해결책 내부를 작업(task) 단위로 분리**하여 제시한다.

   각 해결책의 구조:

   ```
   해결책 N: [한 줄 요약]
   ├─ 작업 N-a [API]    변경 내용 설명    | 영향 파일: ...
   ├─ 작업 N-b [비API]  변경 내용 설명    | 영향 파일: ...
   └─ 작업 N-c [API]    변경 내용 설명    | 영향 파일: ...
   위험도: 낮음 / 중간 / 높음
   ```

   작업 분류 기준:
   - `[API]`: Controller / Service / Repository / DTO / Entity 레이어 변경 포함 (WebSocket·메시징 핸들러 등에서 분리된 Service 포함, 해당 레이어를 직접 지원하는 Mapper·Validator·Helper·Factory·Enum 포함) → Test-First 절차
   - `[비API]`: 설정, 문서, 인프라, 유틸, 메시징 핸들러 라우팅/설정 등 → 일반 코드 변경 절차

4. **사용자 확인 대기** — 적용할 해결책 번호 선택 요청 (예: "1, 3번 적용")

## Phase 2: 의존성 분석 & 실행 계획

사용자가 해결책을 선택하면:

1. 선택된 해결책의 모든 작업을 평탄화(flatten)하여 작업 목록을 만든다.
2. 작업 간 의존성을 분석한다:
   - **같은 파일을 수정**하는 작업 → 순차 처리
   - **같은 도메인의 연관 로직을 변경**하는 작업 → 순차 처리
   - **서로 다른 도메인/파일을 독립적으로 변경**하는 작업 → 병렬 처리 가능

3. 실행 계획을 보고한다:

   ```
   실행 계획:

   [병렬 그룹 A]
   - 작업 1-a [API]   AuthService 만료 검증 로직 수정     → Test-First
   - 작업 1-b [비API] 설정 파일 토큰 만료 시간 변경        → 코드 변경

   [병렬 그룹 B]
   - 작업 3-a [API]   NotificationService 재시도 로직 추가 → Test-First

   [순차 그룹 C] (같은 도메인 변경)
   - 작업 2-a [API]   RefreshTokenService 갱신 메서드 추가 → Test-First [1순위]
   - 작업 2-b [API]   AuthController 갱신 엔드포인트 추가  → Test-First [2순위]

   실행 방식: 그룹 A, B, C를 병렬 실행 / 그룹 내부는 기본 순차 실행 (한 에이전트가 그룹 하나 담당)
   ```

4. 실행 계획 보고 후 사용자 확인 없이 즉시 Phase 3으로 진행한다.

## Phase 3: 이슈 & 브랜치

실행 계획 보고 후, 코드 변경 전에 이슈와 브랜치를 먼저 생성한다.

1. 변경 성격에 맞는 타입 선택:

| 타입 | label | 브랜치 prefix | 이슈 템플릿 |
|------|-------|---------------|-------------|
| `[FEAT]` | `✨ FEAT` | `feat/` | `기능-구현.md` |
| `[FIX]` | `🔧 FIX` | `fix/` | `기능-수정.md` |
| `[BUG]` | `🕷️ BUG` | `bug/` | `오류-수정.md` |
| `[CHORE]` | `🔩 CHORE` | `chore/` | `기타-수정.md` |
| `[REFACT]` | `♻️ REFACT` | `refact/` | `리팩토링.md` |
| `[DOCS]` | `📜 DOC` | `docs/` | `문서-작업.md` |

> 위 label·브랜치 prefix·이슈 템플릿 파일명은 예시다. 프로젝트의 `.github/ISSUE_TEMPLATE/` 및 라벨 컨벤션에 맞게 조정한다.

2. `gh issue create` — 제목: `[TYPE] 설명` (선택된 해결책들을 요약), 본문: 위 이슈 템플릿의 구조를 따르되 Phase 2의 실행 계획 기반으로 내용 채움
3. 브랜치 생성 `"<prefix>#<이슈번호>"` → checkout
4. **Preflight 이월 변경 승격**: `HAS_STASH=true`였다면 워킹 트리의 미커밋 변경을 WIP 커밋으로 올려 워킹 트리를 clean하게 만든다:
   - `git add -A -- ':!.problem/frozen-tests.txt' ':!.problem/local' && (git diff --cached --quiet || git commit -m "wip: preflight carry-over")` — 감사용 동결 목록과 로컬 전용 상태 파일은 커밋에서 제외하고, stash pop 결과가 dev와 동일해 스테이징이 비면 커밋을 스킵한다
   - 이는 병렬 통합(cherry-pick)의 전제조건이다. 더티 워킹 트리 위에서 cherry-pick하면 `error: your local changes would be overwritten`으로 실패하고, 병렬 에이전트 워크트리에는 이 변경이 없어 의미적 충돌이 통합 시점까지 감지되지 않는다.
   - 이 WIP 커밋은 Phase 5에서 최종 커밋에 squash되거나 정리된다.

> **중단 시 정리**: Phase 3에서 이슈/브랜치를 먼저 만들므로, 이후 Phase가 실패하거나 사용자가 중단하면 고아 이슈/브랜치가 남는다. 중단 시 이슈에 중단 사유를 코멘트로 남기고, 사용자에게 브랜치/이슈 삭제 여부를 확인한다.

## Phase 4: 코드 변경 실행

이슈 브랜치에서 코드 변경을 실행한다.

### 사전 단계: 테스트 생성 & 동결 (`[API]` 작업 대상)

**원칙: 심판(테스트)과 선수(구현)를 같은 에이전트가 만들지 않는다.** 같은 에이전트가 둘 다 만들면 스펙 오독이 테스트와 구현 양쪽에 복제되어 서로를 통과시킨다. 테스트는 가능하면 Codex가 스펙에서 독립 파생 작성하고, Claude는 스텁 준비·검증·리뷰·동결만 담당한다. Codex를 쓸 수 없으면 Claude가 테스트까지 작성(Claude 단독 모드)하되, **동결 이하의 절차는 모드와 무관하게 동일하다** — 동결의 가치 절반(구현 루프 중 테스트 변조 방지)은 작성자 분리 없이도 성립하기 때문이다.

> **Codex CLI는 선택 사항이다.** 없으면 테스트 작성자만 Claude로 바뀌며 사전 단계 자체는 스킵하지 않는다.

**모드 결정** — Codex CLI의 설치·인증 상태를 확인하여 **테스트 작성자만** 결정한다 (이후 절차는 두 모드 공통):

1. `which codex`로 설치 여부 확인. **미설치**면 Claude 단독 모드로 진행한다 (사전 단계는 스킵하지 않는다 — 테스트 작성자만 Claude로 바뀐다). 이때 사용자에게 한 줄 알린다: `ℹ️ Codex CLI가 없어 Claude 단독 모드로 진행합니다(동결 포함 절차 동일). (Codex를 쓰려면 설치 후 다시 실행)`
2. 설치돼 있으면 인증 상태를 확인한다: 먼저 `codex login status`로 확인한다 (LLM 호출이 아니라서 과금이 없다 — **exit 0이면 인증 정상**). 이 서브커맨드를 지원하지 않는 구버전 CLI면 무해한 `print OK` 프로브를 사용하되, **`mktemp -d`로 만든 빈 임시 작업 디렉토리 + `-s read-only` sandbox**에서 실행하고 아래 4중 방어와 동일하게 프롬프트 파일의 stdin 리다이렉트·출력 파일 리다이렉트·MCP 비활성화·timeout을 적용한다. 프로브에 프로젝트 경로나 `workspace-write`를 주지 않으며, 결과 확인 후 임시 디렉토리를 삭제한다. positional prompt 프로브는 stdin 추가 입력 대기로 hang할 수 있으므로 금지한다. (실제 테스트 생성 호출을 프로브로 쓰면 실패가 사전 단계 진입 후에야 드러나 순서가 꼬인다.) 인증 확인이 실패하면 **자동 폴백하지 않고 사용자에게 선택을 요청**한다:

   ```
   ⚠️ Codex CLI는 설치돼 있으나 인증이 필요합니다.
   터미널에서 `! codex login`으로 로그인할 수 있습니다.

   진행 방식을 선택하세요:
   1. codex login 완료 — 로그인을 마쳤으면 Codex 모드로 진행
   2. Claude 단독 모드로 진행 — Claude가 테스트를 작성하며, 동결 포함 이하 절차는 동일
   ```
   - **1 선택** → 인증을 재확인한 뒤 Codex 모드로 진행한다. 재확인도 실패하면 다시 이 선택지를 표시한다.
   - **2 선택** → Claude 단독 모드로 진행한다 (사전 단계는 스킵하지 않는다).
3. 인증이 정상이면 Codex 모드로 진행한다.

모든 `[API]` 작업에 대해, 구현 시작 전에 테스트를 먼저 생성·동결한다. 이 단계는 구현(병렬/순차) 실행 전에 완료한다. 구현은 모드 공통으로 `[API] Test-First 구현 절차`를 적용한다.

각 `[API]` 작업마다:

1. **스텁 준비 (Claude)**: 인터페이스 시그니처는 아키텍처 결정이므로 구현측(Claude)이 소유한다 — 테스트 작성자에게 시그니처 설계를 맡기면 코드베이스 컨텍스트가 가장 적은 에이전트에게 설계권이 넘어간다. 신규 계약이 필요하면 Claude가 먼저 컴파일에 필요한 최소 골격만 만든다: concrete 메서드는 미구현 예외 본문, interface/abstract 메서드는 선언만, DTO/record는 필드·생성자 등 타입 골격만 준비한다 — **로직 금지**. 테스트 작성자는 테스트 소스·fixture 외 파일을 수정하지 않는다. **Codex 모드에서는 매 호출·재호출 직전 허용 경로 집합과 모든 기존 미커밋 파일의 실제 내용 사본 + 경로·모드 메타데이터(추적/비추적 모두, hash만 기록 금지)를 `.problem/local/codex-baseline/`에 기준선으로 보존**하고, 호출 후에는 이 기준선 대비 Codex가 새로 만든 변경만 검사한다 — 선행 스텁이나 기존 미커밋 변경을 Codex 위반으로 오인해 되돌리면 안 된다. 기준선은 복구·감사 직후 삭제하며 커밋하지 않는다.

2. **정보 수집**:
   - 스펙: 해당 작업의 변경 내용 (Phase 1에서 확정된 해결책 상세)
   - 테스트 컨벤션: 동일/유사 도메인의 기존 테스트 파일 1~2개 전문 (프로젝트의 테스트 프레임워크·어노테이션·Mock 전략·데이터 셋업/정리 패턴을 담고 있는 것)
   - 인터페이스 시그니처: 변경 대상 Controller/Service/Repository의 현재 메서드 시그니처 + 1번에서 준비한 스텁 시그니처

3. **테스트 작성** — 모드에 따라 작성자만 다르고, 4번 이후 절차는 동일하다.

   **Claude 단독 모드**: Claude가 프로젝트의 기존 테스트 코드를 탐색하여 테스트 프레임워크, 어노테이션, Mock 전략, 데이터 셋업/정리 패턴, 디렉토리 구조를 파악하고 그대로 따라 직접 작성한다:
   - **Controller/웹 계층 테스트**: 하위 레이어(Service 등)를 Mock하고 HTTP 매핑·요청/응답 직렬화·상태코드를 검증하는 가벼운 단위 테스트 패턴을 따른다
   - **Service/도메인 테스트**: Repository·DB는 실제 빈/컨테이너로, 외부 경계(메시징·외부 API 등)만 Mock하는 통합 테스트 패턴을 따른다. 테스트 간 데이터 정리와 Helper를 통한 데이터 생성 컨벤션을 준수한다

   **Codex 모드**: 아래와 같이 Codex를 호출한다. 모델·reasoning effort는 CLI 플래그로 **명시 고정**한다. 전역 config 기본값에 의존하지 않아 팀원별 config 편차로 심판(테스트) 품질이 조용히 열화되지 않게 한다 (CLI 플래그가 config보다 우선). 핀은 lockfile처럼 의존성 버전으로 관리한다 — **핀 모델이 미가용**(로그에 model not found류 에러)이면 기본 모델(`-m`·effort 플래그 제거)로 1회 재시도하고, 동결 커밋 메시지에 사용 모델을 명시한다.

   **⚠️ hang 방지 (필수) — 4중 방어**: codex 호출이 작업 시작 전(입력 대기) 또는 작업 완료 후에 종료되지 못해 Bash 도구가 무한 대기하는 사고를 아래 4가지로 원천 차단한다.
   1. **프롬프트는 positional 인자가 아니라 stdin 리다이렉트로 주입한다 (`< "$CODEX_PROMPT"`)** — 가장 흔한 hang. `codex exec`는 stdin이 파이프로 감지되면 positional 프롬프트를 줘도 stdin을 `<stdin>` 블록으로 **추가로 읽으려** 한다(`codex exec --help`: "If stdin is piped and a prompt is also provided, stdin is appended"). 에이전트 Bash 도구의 stdin은 EOF가 오지 않는 파이프라서, `codex exec ... "프롬프트"`처럼 positional로만 주면 codex가 입력의 끝을 무한 대기하며 hang한다(실측: 로그에 `Reading additional input from stdin...`만 남고 **파일 변경 0건**, Bash 도구 timeout까지 무응답). 프롬프트를 파일에 써서 `< file`로 넘기면 stdin에 프롬프트가 채워지고 EOF도 명확해 codex가 읽는 즉시 진행·종료한다. (heredoc+변수 `read -d '' PROMPT` → `"$PROMPT"`도 결국 positional이라 동일하게 hang하므로 쓰지 않는다.)
   2. **출력은 파이프가 아니라 로그 파일로 리다이렉트한다 (`> "$LOG" 2>&1`)** — codex가 자체 검증 등으로 빌드 도구를 실행하면 **빌드 도구 데몬(예: Gradle daemon)이 codex의 stdout 파일디스크립터를 상속**한다. 이때 codex 출력을 `| tail`처럼 **파이프**로 받으면 codex가 끝나도 데몬이 파이프의 쓰기 끝을 붙잡고 있어 EOF가 오지 않고, 파이프라인이 무한 대기한다(실측: 7분+ 무응답). **파일 리다이렉트는 파이프 자체가 없어 이 hang을 구조적으로 제거**한다 — codex 프로세스가 끝나는 즉시 명령이 반환한다. (`| tail`, `| grep` 등 codex 출력에 대한 파이프 사용 금지.)
   3. **프롬프트에서 codex가 빌드/테스트 등 셸 명령을 실행하지 않도록 지시한다** — 데몬 자체가 안 뜨면 hang 원인이 사라진다. 컴파일/RED 검증(3계층)은 codex가 아니라 **호출측(Claude)이 별도 Bash 호출**로 수행한다.
   4. **`timeout`으로 상한을 둔다** — 위 방어로도 남을 수 있는 프로세스 레벨 hang(일부 환경의 IDE/streamable MCP가 stream을 안 닫는 경우 등)을 강제 종료한다. MCP를 아예 배제하려면 `-c mcp_servers='{}'`(빈 테이블 TOML)로 모든 MCP 서버를 비운다 — 테스트 생성에 MCP는 불필요하다. (IntelliJ IDEA 등 IDE의 MCP가 붙어 있는 환경에서 특히 중요하다. `-c mcp_servers.idea.enabled=false` 같은 개별 필드 override는 존재하지 않는 서버를 겨냥하면 무효과이므로 쓰지 않는다.)

   ```bash
   mkdir -p .problem/local
   CODEX_LOG=.problem/local/codex-out.log       # 로컬 전용(커밋 금지)
   CODEX_PROMPT=.problem/local/codex-prompt.txt # 프롬프트 파일(로컬 전용) — 아래 템플릿에 수집 항목을 채워 파일로 먼저 기록한다
   # timeout은 있을 때만 상한 래핑(권장). macOS 기본엔 timeout/gtimeout이 없으므로 없으면 그냥 실행 —
   # stdin 리다이렉트 + 파일 출력 + 빌드 금지로 입력/출력 파이프 hang은 구조적으로 차단돼 있고, 최종 backstop은 Bash 도구 timeout이다.
   TO=""; command -v timeout >/dev/null && TO="timeout 480"; command -v gtimeout >/dev/null && TO="gtimeout 480"
   $TO codex exec \
     -C "$CLAUDE_PROJECT_DIR" \
     -s workspace-write \
     --ephemeral \
     -m gpt-5.6-sol \
     -c model_reasoning_effort="xhigh" \
     -c mcp_servers='{}' \
     --color never \
     < "$CODEX_PROMPT" \
     > "$CODEX_LOG" 2>&1
   ec=$?; echo "codex exit=$ec"; tail -40 "$CODEX_LOG"
   ```
   - **프롬프트는 `$CODEX_PROMPT`에 먼저 기록한 뒤 `< "$CODEX_PROMPT"`로 주입한다** — positional 인자(`codex exec ... "프롬프트"`)나 heredoc 변수(`"$PROMPT"`)로 주면 stdin 파이프 감지로 hang한다(방어 1).
   - 이 Bash 호출은 **codex 출력에 파이프를 걸지 않는다**(`tail`은 codex가 아니라 **로그 파일**을 읽으므로 안전). 포그라운드로 돌리되 Bash 도구 `timeout`은 크게(예: 540000ms) 잡아, `timeout` 미설치 환경에서도 도구 레벨이 최종 backstop이 되게 한다. codex 작업이 길거나 hang이 재차 우려되면 Claude Code Bash의 **`run_in_background: true`**로 실행하고 완료 통지 후 로그·파일을 점검한다 (`&`로 셸 프로세스만 분리하지 않는다).
   - `ec`가 124면 `timeout`이 hang을 끊은 것, Bash 도구 timeout으로 끊겼으면 도구가 알려준다 — 어느 쪽이든 로그와 생성 파일을 점검한다(파일은 이미 디스크에 있을 수 있다). 0이면 정상 종료.
   - 호출 직후 **해당 빌드 도구가 데몬을 사용하고 stop 명령을 제공하는 경우에만** 그 명령(예: Gradle의 `./gradlew --stop`)으로 codex가 띄웠을 수 있는 잔여 데몬을 정리한 뒤 3계층 검증으로 넘어간다.

   **프롬프트 템플릿**:
   ```
   ## 작업 스펙
   {스펙}

   ## 테스트 컨벤션 (기존 예시)
   {기존 테스트 파일 전문}

   ## 현재 인터페이스 시그니처
   {시그니처 — Claude가 준비한 스텁 시그니처 포함}

   ## 지시사항
   위 스펙을 만족하는 테스트 코드만 작성하라. 컴파일에 필요한 스텁은 이미 프로덕션 코드에 준비되어 있다.
   - 테스트 컨벤션의 패턴(어노테이션, Mock 전략, 그룹화 구조, 데이터 셋업/정리 Helper)을 그대로 따른다
   - **테스트 소스와 테스트 fixture 외의 어떤 파일도 생성·수정하지 않는다** — 스텁·프로덕션 코드는 호출측이 관리한다. 테스트가 명세다
   - 구현 로직을 검증하는 신규·변경 테스트 메서드는 **구현 전 기준선**(신규 계약은 준비된 스텁, 기존 계약은 현재 구현)에서 실패해야 한다. 프레임워크 validation처럼 구현과 독립적인 테스트는 기준선에서 통과할 수 있지만 호출측이 메서드별 근거를 기록해야 한다
   - **빌드/테스트 등 어떤 셸 명령도 실행하지 말고 파일 작성만 하라.** 컴파일·RED 검증은 호출측이 수행한다. (codex가 빌드 도구를 띄우면 데몬이 stdout을 상속해 프로세스 종료가 막힌다.)
   - 목표 상태: 테스트 소스 컴파일은 통과(컴파일 GREEN)하고, 구현 의존적인 신규·변경 테스트는 assertion 또는 미구현 예외로 실패(실행 RED)하는 상태
   ```

4. **동결 게이트 — 3계층 기계 검증** — 테스트 작성이 끝난 뒤 **Claude가 별도 Bash 호출로** 수행한다. 모드 공통. (Codex 모드에서 이 빌드 실행은 codex 파이프와 무관한 독립 프로세스라 hang 위험이 없다 — 검증을 codex에게 맡기지 않는 이유가 이것이다.) 아래 세 계층을 모두 통과해야 "동결 가능한 테스트 산출물"로 인정한다.
   1. **파일/Discovery**: 이번 단계에서 생성·수정한 테스트 소스와 fixture를 모두 식별하고, 대상 테스트 클래스가 러너에 실제로 발견되는가 (대상 테스트 실행이 클래스를 못 찾으면 시끄럽게 실패해야 한다)
   2. **컴파일 GREEN**: 테스트 소스 컴파일 명령 통과 (스텁 포함, 테스트 소스 전체가 컴파일되는가)
   3. **실행 RED/정당화 (테스트 메서드 단위)**: 대상 테스트 클래스들을 실행해 이번 단계의 **신규·변경 테스트 메서드 결과를 개별 열거**한다. 구현 의존 메서드는 구현 전 기준선에서 assertion 실패 또는 미구현 예외로 RED여야 한다. 기존의 변경하지 않은 메서드는 GREEN이어도 무관하고, 신규·변경 메서드가 GREEN이면 프레임워크 validation 등 구현과 독립적인 이유를 건별로 기록해야 한다 — 정당화되지 않은 GREEN은 "아무것도 안 하는 구현"을 죽이지 못하는 공허 테스트 후보이므로 재작성한다. 컴파일 에러·빈 설정 오류로 실패하면 RED가 아니라 broken이다.
   - 재작성: Codex 모드는 프롬프트를 조정해 재호출, Claude 단독 모드는 직접 수정한다. 해소되지 않으면 사용자에게 보고한다.

5. **정성 리뷰 (Claude)** — 3계층은 기계 검증이라 커버리지·가정 오류를 못 본다. 구현자 관점 개입 금지, 다음만 검사한다:
   - ① 스펙(수용 기준) 전 항목이 테스트로 커버되는가
   - ② 4번에서 정당화된 것 외에 공허 통과 가능성이 있는가
   - ③ 스펙에 없는 가정을 추가했는가
   - ④ (Codex 모드) 호출 직전 기준선·허용 경로와 호출 후 상태를 비교하여 Codex가 지정된 테스트 소스·fixture 외 파일이나 관련 없는 테스트 산출물을 새로 변경하지 않았는지 확인 — 위반 시 **Codex가 만든 delta만** 되돌리고 선행 스텁·기존 변경은 보존. 검사가 끝나면 평문 코드·시크릿을 담을 수 있는 `.problem/local/codex-baseline/`을 즉시 삭제한다 (중단 시 복구 완료 직후 삭제)
   - Codex 모드에서 문제 발견 시 프롬프트를 보강해 **Codex에 재작성 요청**한다 — Claude가 assertion을 직접 고치지 않는다 (컴파일 오류 등 기계적 수정만 허용하며 동결 커밋 메시지에 기록). Claude 단독 모드의 자기 리뷰는 독립성이 없으므로 4번 기계 게이트가 주 방어선이고, 이 리뷰는 커버리지·가정 점검에 집중한다.

모든 `[API]` 작업의 테스트 생성 완료 후 아래를 수행한다. 선택된 작업이 전부 `[비API]`이면 6~8번을 스킵하여 동결 목록·커밋을 만들지 않는다.

6. **동결 목록 기록**: 이번 테스트 작성 단계에서 **생성·수정한 모든 테스트 소스와 fixture**의 경로를 **리포 루트 기준 상대 경로**로 `.problem/frozen-tests.txt`에 한 줄씩 기록한다 (예: `src/test/java/.../ServiceTest.java`, `src/test/resources/.../fixture.json`). 절대 경로로 기록하면 워크트리(다른 루트 디렉토리)에서 훅이 매칭하지 못한다. 스텁 파일은 구현 대상이므로 목록에 넣지 않는다. 기록 즉시 PreToolUse 훅이 해당 테스트 산출물에 대한 Edit/Write를 차단한다. (훅은 **Edit/Write 도구 호출만** 차단하며 git 등 Bash 명령은 차단하지 않는다 — 따라서 6번과 7번 사이에 훅이 이미 활성이어도 7번의 `git add`/`git commit`은 정상 동작한다.)
7. **동결 커밋**: `git add <테스트 소스·fixture> <스텁 파일들> && git add -f .problem/frozen-tests.txt && git commit -m "test: RED 테스트 산출물 + 스텁 — 동결 (작성: Codex)"` — Claude 단독 모드면 `(작성: Claude 단독)`으로, 기본 모델 폴백·기계적 수정이 있었으면 그 사유도 메시지에 병기한다. fixture와 스텁까지 함께 커밋해야 병렬 워크트리에서도 재현·컴파일 가능한 커밋이 된다. **`frozen-tests.txt`를 강제 추가해 함께 커밋하는 것이 Phase 5 감사의 전제다**: 구버전 설치 안내로 상태 디렉토리가 ignore된 환경에서도 감사 기준을 남기며, 워킹 트리의 목록은 훅용(변조 가능), 커밋된 목록은 감사용(불변 기준)으로 역할이 분리된다.
8. **동결 커밋 SHA 기록**: `git rev-parse HEAD > .problem/local/frozen-commit.txt` — cherry-pick으로 히스토리가 늘어난 뒤에도 Phase 5 감사가 동결 커밋을 찾는 기준이다. 이 파일은 **로컬 전용이며 커밋하지 않는다** — 감사 기준 SHA는 메인 세션만 읽으므로 커밋할 이유가 없고, TEST-DISPUTE 재동결 시 로컬에서만 갱신된다.

### 실행 모드 결정

- **그룹이 2개 이상**: 모든 그룹을 에이전트로 병렬 실행한다 — 순차 그룹도 에이전트 하나가 통째로 담당한다(그룹 내부는 순차). **Phase 4 통합 완료 전까지** 메인 세션은 코드를 직접 수정하지 않고 **통합(cherry-pick)만 담당**한다 — 메인 세션이 순차 작업으로 워킹 트리를 더럽힌 상태에서 cherry-pick이 겹치는 경로를 원천 차단하기 위함이다. (통합 완료 후에는 충돌 우려가 사라지므로 디스퓨트 해소·Phase 5 수정 루프에서 메인 세션이 코드를 수정한다.)
- **그룹이 1개**: 워크트리 없이 메인 세션에서 순차 실행한다 (아래 「순차 실행」).

### 병렬 실행 (그룹 2개 이상)

각 그룹을 **Agent tool로 동시에 호출**한다.

**워크트리 라이프사이클**:
- **base**: 각 워크트리는 반드시 **현재 이슈 브랜치의 HEAD**를 base로 생성한다. `[API]` 동결이 있으면 테스트 산출물 + 스텁 커밋이, Phase 3 WIP가 있으면 그 커밋이 HEAD에 포함된다. base가 다르면 cherry-pick 시 선행 커밋이 중복/누락된다.
- **네이밍**: 각 에이전트는 워크트리 내에서 임시 브랜치 `wt/<이슈번호>-<그룹>`(예: `wt/123-A`)에서 작업한다. 임시 브랜치명에는 `#`을 넣지 않는다 — 따옴표 방어가 필요한 표면을 줄인다.
- **정리**: 통합 완료 후 각 워크트리를 `git worktree remove`하고 임시 브랜치를 삭제한다. 누락하면 실행할 때마다 워크트리/브랜치가 누적된다.

각 에이전트 prompt에는 다음을 포함한다:
- 해당 그룹에 속한 작업들의 상세 내용 (변경 대상, 변경 방법)
- 각 작업의 타입에 따른 실행 절차 (`[API]`: `[API] Test-First 구현 절차` / `[비API]`: `[비API] 일반 절차` 전문)
- **그룹 내부는 기본 순차 실행**한다 (한 에이전트가 그룹 하나를 받음). 순차 작업이 있으면 순서를 명시한다.
- **커밋 소유권**: 각 작업 완료 후 워크트리 내에서 **로컬 커밋**까지 수행한다. 완료 보고 시 **생성한 커밋 SHA 목록을 순서대로 반드시 포함**한다 (메인 세션이 cherry-pick 대상을 수집하는 유일한 경로). PR 생성이나 push는 하지 않는다.
- **TEST-DISPUTE 프로토콜 (병렬)**: 에이전트는 사용자 확인을 대기할 수 없으므로, 디스퓨트 발생 시 해당 작업의 구현을 중단하고(커밋하지 않음), **디스퓨트된 작업에 의존하는 그룹 내 후속 작업도 함께 중단**한다 — 미구현 스텁이나 기존 결함 상태 위에 후속 작업을 구현하면 잘못된 전제 위의 구현이 되어 해소 후 재작업 위험이 있다. **독립적인 나머지 작업만** 계속 진행한다. 완료 보고에 `⚠️ TEST-DISPUTE` 항목(테스트 메서드, 스펙 근거, 판단 이유)과 **중단된 작업 목록**을 포함한다. 후속 처리(사용자 확인, 재동결 또는 구현 수정, 중단된 의존 작업들의 구현)는 **통합 절차 완료 직후, Phase 5 진입 전에** 메인 세션이 수행한다 — Phase 5의 전체 테스트는 디스퓨트 작업의 동결 테스트를 구현 전 기준선에서 실행하므로, 해소가 선행되어야 빌드가 의미 있다 (`[API]` Test-First 구현 절차 4번 참조).
- **무거운 통합 테스트 경합 방지**: N개 워크트리가 동시에 빌드 데몬 + DB 컨테이너(예: Testcontainers) 등을 띄우면 로컬 머신에서 메모리/포트 경합이 발생한다. 따라서 병렬 에이전트는 **DB 컨테이너 등 외부 리소스를 띄우는 무거운 통합 테스트를 실행하지 않는다** — 구현 + 컴파일 검증(프로덕션+테스트 소스)까지만 수행한다. 단, **외부 리소스 없는 가벼운 단위 테스트(예: Controller 웹 계층 단위 테스트)는 워크트리에서 실행해 GREEN을 확인한다**. 통합 테스트의 실제 실행은 cherry-pick 통합 후 메인 세션에서 일괄 수행한다(아래 통합 절차 및 Phase 5).
- **동결 목록 (`[API]` 동결이 있는 경우)**: 워크트리는 동결 커밋을 포함한 브랜치 HEAD에서 생성되므로 `.problem/frozen-tests.txt`가 자동으로 포함된다 — 별도 복사가 필요 없다. 상대 경로로 기록돼 있으므로 워크트리 루트 기준으로 훅이 정상 매칭한다.

**통합 절차**:
1. 모든 에이전트 완료 후, 각 에이전트가 보고한 커밋 SHA를 순서대로 수집한다.
2. 현재 브랜치에서 SHA들을 **cherry-pick**으로 통합한다. 그룹 간 적용 순서는 **그룹 알파벳 순(A → B → C)**, 그룹 내부는 에이전트가 보고한 커밋 순서를 따른다 — 독립 그룹이라 이론상 순서 무관하지만, 충돌 해결의 재현성과 디버깅 용이성을 위해 순서를 고정한다. 충돌 발생 시 메인 세션이 해결한 뒤 진행한다.
3. 통합 직후 컴파일 검증(프로덕션+테스트 소스)으로 통합 컴파일을 검증한다 (전체 테스트 실행은 Phase 5).
4. 각 워크트리 `git worktree remove` + 임시 브랜치 삭제.

### 순차 실행 (그룹 1개)

작업들을 **순서대로 현재 세션에서 실행**한다.
- 각 작업의 타입에 따른 절차를 따른다 (아래 참조).
- 각 작업 적용 후 컴파일 검증(프로덕션+테스트 소스)으로 이전 작업과의 호환성을 확인한다 (프로덕션 아티팩트만 만드는 빌드 명령은 테스트 소스를 컴파일하지 않아 동결 테스트의 컴파일 깨짐을 잡지 못한다. Phase 4 반복 구간에서는 `clean` 없이 인크리멘털 빌드를 유지한다).
- 단, PR은 하지 않는다. **최종 커밋 시점**: 순차 작업만 있는 실행에서는 Phase 5 종료 후 Phase 6(`pipeline.md`)이 최종 커밋을 담당한다. Phase 5까지는 미커밋 상태로 유지된다.

### `[API]` Test-First 구현 절차

> 사전 단계에서 동결된 테스트가 이미 존재한다 (작성자는 모드에 따라 Codex 또는 Claude — 이 절차는 모드 공통). 해당 테스트를 GREEN으로 만드는 구현에만 집중한다.

1. **동결 테스트 확인**: 해당 작업의 동결 테스트 소스와 fixture를 읽고, 기대하는 인터페이스와 동작을 파악한다

2. **구현 코드 작성**: 스펙 + 동결 테스트 산출물을 모두 참조하여, 사전 단계에서 준비한 스텁 본문을 구현한다. 구현 중 새 시그니처가 필요하다고 드러나면 테스트에 맞춰 임의 생성하지 않는다 — 작업 스펙과 Claude 소유의 시그니처/스텁을 먼저 갱신하고, 영향받는 테스트를 다시 파생·검증·재동결한 뒤 구현을 재개한다.

3. **테스트 실행 & 검증**:
   - **병렬 워크트리 실행 시**: 가벼운 단위 테스트는 실행해 GREEN을 확인한다. DB 컨테이너 등 외부 리소스를 띄우는 무거운 통합 테스트는 경합 방지를 위해 컴파일 검증(프로덕션+테스트 소스)까지만 수행하고 실행은 생략한다 — 실제 실행/검증은 통합 후 메인 세션(Phase 5)에서 일괄 수행한다.
   - **순차/단독 실행 시**: 변경된 도메인의 테스트 클래스만 실행 (Controller, Service 테스트가 모두 있으면 둘 다 실행)
   - 테스트 실패 시 → **구현 코드를 수정**하여 재실행 (테스트는 동결 상태이므로 수정 불가)

4. **이의 제기** (해당 시에만): **기본 추정은 구현 결함이다** — 동결은 "테스트가 맞다"는 추정이므로 입증 책임은 항상 구현 쪽에 있다. 그럼에도 디스퓨트가 성립하면 3갈래로 분류한다: **구현 결함**(구현 수정 — 기본값) / **테스트 결함**(테스트가 스펙 오독) / **스펙 결함**(Phase 1 해결책 자체가 모호·모순 — 사용자 확인이 곧 스펙 확정 행위다):

   **순차/메인 세션 실행 시**:
   - `⚠️ TEST-DISPUTE` 라벨과 함께 해당 테스트 메서드, 스펙 근거, 판단 이유, 분류(테스트 결함/스펙 결함)를 사용자에게 보고한다
   - **사용자 확인 대기** — 구현 결함이면 구현 수정을 지시받아 조정한다. 테스트 결함이면 확정된 해석에 따른 테스트 재파생을 승인받는다. 스펙 결함이면 모호·모순인 수용 기준의 해석을 사용자에게 선택·확정받아 Phase 1 해결책 기록을 갱신한 뒤 테스트를 재파생한다.
   - 테스트/스펙 결함 승인 시 관련 테스트 산출물만 `.problem/frozen-tests.txt`에서 제거 후 수정하고 **수정 완료 즉시 목록에 다시 추가하여 재동결**한다(노출 시간 최소화).
   - 테스트 수정은 **가능하면 Codex에 재파생을 요청**한다 (확정된 해석을 프롬프트에 반영해 재호출 — 심판/선수 분리 유지). Codex 불가 시 Claude가 수정하되 **사용자가 승인한 변경 내용에 한정**한다.
   - **재동결 시 새 동결 커밋을 만들고 감사 기준을 갱신한다**: `git add <수정된 테스트 소스·fixture> && git add -f .problem/frozen-tests.txt && git commit -m "test: TEST-DISPUTE 승인 반영 — 재동결 (<판단 사유 요약>)"` 후 `git rev-parse HEAD > .problem/local/frozen-commit.txt`. **판단 사유를 커밋 메시지에 남긴다** — 승인 이력과 근거가 히스토리에 남아야 사후 추적이 가능하다. 감사 기준은 항상 **최신 동결 커밋**이 되어 승인된 수정이 Phase 5 감사에서 변조로 오탐되지 않는다.

   **병렬 워크트리 실행 시** (에이전트는 실행 도중 사용자 입력을 받을 수 없다):
   - 에이전트는 **사용자 확인을 대기하지 않는다**. 디스퓨트가 발생한 작업의 구현을 그 시점에 중단하고(해당 작업은 커밋하지 않는다), **그 작업에 의존하는 그룹 내 후속 작업도 함께 중단**한다. 독립적인 나머지 작업만 계속 진행하며, **완료 보고에 `⚠️ TEST-DISPUTE` 항목**(테스트 메서드, 스펙 근거, 판단 이유)과 **중단된 작업 목록**을 포함한다.
   - 메인 세션이 나머지 커밋을 통합한 뒤 — **통합 절차 완료 직후, Phase 5 진입 전에** — 사용자 확인을 받고, 승인 결과에 따라 **메인 세션에서** 위 재동결 절차(테스트 수정 승인 시) 또는 해당 작업의 구현(구현 수정 지시 시)을 수행하고, **중단된 의존 작업들의 구현**도 이어서 수행한다. 디스퓨트된 작업의 미완성 커밋이 cherry-pick되는 일이 없도록 하는 것이 이 프로토콜의 목적이다.

5. **빌드 검증**: 컴파일 검증(프로덕션+테스트 소스)

### `[비API]` 일반 절차

1. 변경사항 코드에 적용
2. 컴파일 검증(프로덕션+테스트 소스) 실행
3. **기존 테스트 영향 검증**:
   - 변경된 클래스를 의존하는 기존 테스트가 있는지 확인한다 (특히 웹 계층 테스트에서 자동 로딩되는 전역 예외 핸들러·Filter·Interceptor 등의 변경 시 주의)
   - 영향받는 테스트가 있으면 해당 테스트를 실행한다
   - 테스트 실패 시 테스트 코드를 수정한다 (예: 새 의존성 추가로 인한 Mock 누락 등)
   - 단, 영향이 **동결된 테스트**(`.problem/frozen-tests.txt` 목록)에 미치는 경우 수정하지 말고 `⚠️ TEST-DISPUTE`에 준해 사용자에게 보고한다 (훅이 Edit/Write를 차단하므로 어차피 직접 수정할 수 없다)
   - 영향받는 테스트가 없으면 스킵한다
4. 모든 변경 파일 재검토: 문법/컴파일 에러, import 누락, 호환성, 보안 취약점

## Phase 5: 통합 & 검토

1. 모든 작업 적용 완료 후 전체 변경사항을 보고한다:
   - 작업별 변경 파일 목록
   - 테스트 결과 요약 (`[API]` 작업의 경우)
2. **최종 빌드 & 전체 테스트 검증**: 프로젝트의 전체 빌드 명령이 테스트 실행을 포함한다면 그 명령 하나로 전체 테스트가 함께 돌아간다 — 별도의 테스트 단계를 중복으로 두지 않는다. Phase 5에서만 `clean`(캐시 폐기)을 붙인다.
   - **전체 테스트가 필요한 경우** (병렬 통합이 있었음 / `[비API]` 작업에서 기존 테스트를 수정했음): `clean` 포함 전체 빌드+테스트 **1회**. 특히 병렬 모드에서는 무거운 통합 테스트를 이 시점에 **처음 일괄 실행**하는 것이므로(Phase 4 병렬 구간에서는 통합 테스트를 컴파일까지만 수행) 반드시 이 경로를 탄다.
   - **스킵 가능한 경우** (순차 단독 실행 + 테스트 수정 없음): Phase 4에서 테스트 검증이 완료되었으므로 테스트를 제외한 빌드만 검증한다.
   - **테스트 실패는 이 시점의 정상 경로다**: 병렬 에이전트는 자기 구현이 GREEN인지 확인한 적이 없다(무거운 통합 테스트 미실행). 실패 시 사용자에게 떠넘기고 멈추지 않는다 — 메인 세션이 **동결을 유지한 채 구현 코드만 수정**하고 재실행하는 루프를 GREEN이 될 때까지 반복한다. 테스트가 스펙을 잘못 해석했다고 판단되는 경우에만 `⚠️ TEST-DISPUTE` 절차(Phase 4 참조)를 준용해 사용자 확인을 받는다.
   - **루프 내 재실행은 실패분만 대상**으로 한다(실패한 테스트 클래스만 실행). 반복마다 `clean` 전체 빌드(전체 통합 스위트 + 캐시 폐기)를 돌리면 Phase 4에서 제거한 중복이 루프 형태로 재도입된다. 실패분이 전부 GREEN이 된 뒤 **최종 확인으로 `clean` 포함 전체 빌드+테스트를 1회 재실행**한다.
   - **서킷 브레이커**: 동일 테스트가 **3회 연속** 실패하면 TEST-DISPUTE 여부와 무관하게 루프를 멈추고, 사용자에게 현황(실패 테스트, 시도한 수정, 실패 원인 분석)을 보고한 뒤 지시를 대기한다.
3. **동결 무결성 사후 감사**: 동결이 있었던 경우:
   - **감사는 항상 마지막 코드 변경 뒤에 온다**: 코드 변경이 발생하는 모든 단계(2번의 수정 루프, TEST-DISPUTE 처리)가 완료된 뒤 실행하고, 감사 이후 코드가 다시 변경되면(6번의 추가 수정 요청 등) 감사를 **재실행**한다. 감사를 수정 루프 앞에 두면, 우회 편집의 유혹이 가장 강한 구간(RED가 풀리지 않는 수정 루프)이 무방비가 된다.
   - 동결 커밋 SHA를 `.problem/local/frozen-commit.txt`에서 읽는다. 이 SHA는 항상 **최신 동결 커밋**이다 — TEST-DISPUTE 승인으로 테스트를 수정했다면 재동결 시점에 이 파일이 갱신되어 있으므로(Phase 4 참조), 승인된 수정이 변조로 오탐되지 않는다.
   - 감사 기준 목록은 워킹 트리 파일이 아니라 **커밋된 원본**에서 복원한다: `git show <동결 커밋>:.problem/frozen-tests.txt`. 워킹 트리 목록을 기준으로 삼으면 목록에서 줄을 지우는 것만으로 해당 파일이 감사 범위에서 조용히 빠진다 — 커밋된 원본이 이 순환(감사가 감사 대상 목록의 무결성에 의존)을 끊는다.
   - `git diff <동결 커밋> -- <복원한 목록의 테스트 소스·fixture>`를 실행하여 동결 테스트 산출물이 변경되지 않았는지 확인한다. PreToolUse 훅은 Edit/Write만 막으므로 Bash(`sed -i`, `cat >` 등) 우회 편집은 이 감사로만 탐지된다. diff가 비어있지 않으면 사용자에게 보고하고 원인을 확인한다. (훅=예방, diff=감사 — 두 층이 필요하다.)
   - 워킹 트리의 `.problem/frozen-tests.txt`가 커밋된 원본과 다르면 그 자체도 이상 신호이므로 함께 보고한다.
4. **Preflight WIP 커밋 정리**: Phase 3에서 `wip: preflight carry-over` 커밋을 만들었다면, 최종 커밋 시점(Phase 6)에 해당 WIP 커밋이 실제 작업 커밋에 squash되도록 정리 방침을 `pipeline.md`에 전달한다. 동결이 있었던 경우, **최종 커밋에서 `.problem/frozen-tests.txt`를 인덱스에서만 제거**하는 방침도 함께 전달한다 — `automation/pipeline.md` Step 1은 매 실행·재시작마다 `git rm --cached --ignore-unmatch` 후 해당 파일과 `.problem/local/`을 `git add -A`에서 제외한다. 따라서 상태 파일은 PR/dev에 남지 않으면서 워킹 트리에는 배포 성공까지 남아 훅 보호를 유지한다. (`frozen-commit.txt`는 `.problem/local/` 하위 로컬 전용이라 애초에 인덱스에 없다.)
5. **환경변수 체크**: 새 환경변수가 도입된 경우, `.claude/common/pipeline.md`의 「환경변수 추가 체크리스트」를 확인한다.
6. **사용자 확인 대기** — 추가 수정 요청 시 반복. OK 시에도 **동결은 아직 해제하지 않는다**: Phase 6 배포가 실패해 코드를 다시 고치는 시나리오에서 테스트가 무방비가 되지 않도록, `.problem/frozen-tests.txt` 삭제(동결 해제)는 **Phase 6 배포 성공 후**로 미룬다.

## Phase 6: 배포

사용자 확인 완료 시, `.claude/common/pipeline.md`를 읽고 해당 절차를 실행한다.
Preflight는 스킵하고 Step 1부터 시작한다.
CI/CD·PR 체크 대기 폴링은 pipeline.md 「폴링 실행 규칙」에 따라 **`run_in_background: true` Bash로 실행**한다 (포그라운드 `sleep`·Monitor 도구 금지 — 포그라운드 sleep 차단 및 Monitor 중복 프롬프트 방지).

전달할 컨텍스트:
- 브랜치명, 타입, 이슈번호
- 변경사항 요약 (Phase 5에서 보고한 내용)
- Phase 5의 WIP 커밋 정리 방침 및 frozen 상태 파일 인덱스 제거 방침 (있는 경우)

**배포 성공 후**: 배포가 성공적으로 완료되면 그때 `.problem/frozen-tests.txt`·`.problem/local/frozen-commit.txt`가 존재하면 삭제(동결 해제)한다. 배포 실패로 코드를 재수정하는 경우 동결이 유지되어 테스트가 계속 보호된다. 이와 함께 `.problem/local/preflight-stash.patch`와 비정상 중단으로 남은 `.problem/local/codex-baseline/`이 존재하면 삭제한다 — 배포 성공 시점엔 작업이 이미 커밋/푸시되어 이 평문 백업들은 잉여가 된다(단 `preflight-stash.patch`는 배포 실패 시 데이터 손실 방지 안전망으로 남겨둔다).

> **`.problem/local/` 정리 원칙 (로그 vs 데이터 스냅샷)**: 파일이 **재생성 가능한 로그**(`codex-out.log`·`codex-prompt.txt` — codex 실행 흔적, 고정 파일명이라 다음 실행 때 덮어써짐, 비민감)이면 **삭제하지 않고 유지**한다 — 사후 부검(예: codex가 짠 RED 테스트가 이상할 때 근거 추적)에 쓰이고 성장 비용이 없다. 파일이 **실제 내용을 담은 데이터 스냅샷**이면 목적을 다하는 즉시 삭제한다: `codex-baseline/`은 각 Codex 호출의 delta 검사·복구 직후, `preflight-stash.patch`는 배포 성공 후 삭제한다. `frozen-*`는 배포 실패 시 테스트 보호를 위해 성공 후에만 삭제하는 별도 규칙이다. 새 상태 파일이 늘면 이 기준으로 분류한다.

## 파괴적 DB 마이그레이션 2단계 배포

`.claude/common/pipeline.md`의 「파괴적 DB 마이그레이션 2단계 배포 원칙」을 따른다.

**Phase 1 적용**: 해결책에 파괴적 마이그레이션이 포함되면, 작업을 `[1차 배포]`와 `[2차 배포]`로 분리하여 제시한다.

```
해결책 N: ...
[1차 배포 — 이번 워크플로우]
├─ 작업 N-a [API]   앱 코드에서 기존 스키마 참조 제거
└─ 작업 N-b [비API] additive 마이그레이션 (있는 경우)
[2차 배포 — 1차 안정 확인 후 별도 워크플로우]
└─ 작업 N-c [비API] 파괴적 마이그레이션 (DROP/RENAME 등)
```

- Phase 2~6에서는 `[1차 배포]` 작업만 실행한다. `[2차 배포]` 작업은 실행하지 않는다.

## 규칙

- 해결책 제시 시 코드 변경 없이 분석만 수행한다
- 사용자가 선택하지 않은 해결책은 절대 적용하지 않는다
- 하나의 해결책 안에서도 `[API]`/`[비API]` 작업을 반드시 분리하여 제시한다
- `[API]` 작업의 테스트 실패 시 구현 코드를 수정한다 (테스트가 명세이므로 테스트를 수정하지 않는다 — **기본 추정은 구현 결함, 입증 책임은 구현 쪽**). 테스트가 스펙을 잘못 해석했다고 판단되면 `⚠️ TEST-DISPUTE`로 이의를 제기하고 사용자 확인을 대기한다.
- 동결 **커밋** 단위는 "이번 단계에서 생성·수정한 테스트 소스·fixture + 컴파일용 스텁"이며 **모드 공통이다** — Claude 단독 모드에도 동결이 적용된다: 작성자 분리가 없어도 구현 루프 중 테스트 변조 방지 가치는 성립한다. 스텁은 Claude(구현측)가 준비하고 테스트 작성자는 허용된 테스트 소스·fixture 외 파일을 수정하지 않는다. concrete 메서드는 미구현 예외 본문, interface/abstract 메서드는 선언, DTO/record는 최소 타입 골격으로 준비해 컴파일 GREEN과 구현 의존 테스트 RED를 유지한다. 스텁은 구현 대상이므로 보호 목록에는 넣지 않는다.
- 워크플로우 상태 파일 규칙: `.problem/frozen-tests.txt`는 커밋 대상(감사 기준, 최종 커밋에서 인덱스 제거), `.problem/local/` 하위(`frozen-commit.txt`, 기준선 스냅샷, `preflight-stash.patch` 등)는 로컬 전용(커밋 금지). `git add -A`류 일괄 스테이징에는 항상 `-- ':!.problem/frozen-tests.txt' ':!.problem/local'`을 붙인다.
- Phase 4의 코드 변경 중 빌드 검증은 컴파일 검증(프로덕션+테스트 소스)을 사용하고, `clean`을 붙인 전체 빌드는 Phase 5에서만 수행한다 (기본 1회; 수정 루프가 돌았던 경우 루프 종료 후 최종 확인 1회 추가 — 루프 내부에서는 실패한 테스트 클래스만 재실행한다).
- 기존 테스트 패턴(어노테이션, Mock 전략, Helper 구조)을 반드시 따른다 (Codex 모드에서는 프롬프트에 컨벤션으로 전달)
- 병렬 실행 시 각 워크트리는 완전히 독립적으로 작업한다
- 모든 `gh`/`git` 명령 실패 시 에러 내용 사용자에게 보고
- 이슈/PR 생성 시 GitHub 템플릿 형식 준수
- 배포 파이프라인(Phase 6)의 규칙은 `.claude/common/pipeline.md`를 따른다
